## Quick Reference

### What This System Does

The Email Notification System is ConvexPress's transactional email infrastructure, replacing WordPress's `wp_mail()` function, email notification hooks, and PHPMailer transport. It composes, queues, sends, tracks, and retries all outbound emails generated by CMS events: welcome messages, password resets, comment notifications, content publishing alerts, security warnings, and administrative digests.

ConvexPress replaces WordPress's fire-and-forget PHPMailer/SMTP stack with **Resend** (a modern transactional email API) and a **persistent queue** backed by the Convex database. Every email is stored, tracked, and retryable. Delivery status is visible in the admin panel.

### Key Concepts

| Concept | Description |
|---------|-------------|
| **Email Queue** | Persistent Convex table where every email is stored before sending. Enables retry, tracking, auditing. |
| **Email Templates** | 25 pre-defined email templates stored in the database, customizable via admin UI. |
| **Template Variables** | Curly-brace syntax (`{variable_name}`) for dynamic content replacement. |
| **Delivery Priority** | Three modes: `immediate` (send now), `batched` (send within 15 min window), `digest` (weekly aggregation). |
| **Recipient Types** | Four types: `customer`, `employee`, `admin`, `custom` -- resolved to actual email addresses. |
| **Unsubscribe Categories** | Per-category email opt-out: `content`, `comment`, `security`, `system`, `digest`, `all`. |
| **Security-Critical Emails** | Password reset, password changed, account deactivated, deletion confirmation, failed logins -- cannot be unsubscribed. |
| **Rate Limiting** | Per-minute (default 50) and daily (default 1000) email caps to prevent floods. |
| **Exponential Backoff** | Failed emails retry at 5s, 10s, 20s intervals (max 3 attempts). |
| **Event Listeners** | 19 event listener registrations via bootstrap (some handlers trigger multiple templates), covering all 23 non-digest email templates. |

### ConvexPress vs WordPress

| Aspect | WordPress | ConvexPress |
|--------|-----------|-------------|
| **Transport** | PHPMailer over SMTP (`wp_mail()`) | Resend HTTP API via Convex actions |
| **Template Engine** | Inline PHP string concatenation | Database templates with admin editor + `{variable}` syntax |
| **Queue** | None (sends synchronously during request) | Persistent Convex `emailQueue` table |
| **Delivery Tracking** | None (fire-and-forget) | Full delivery status tracking per email |
| **Retry** | None (if `wp_mail()` fails, it's lost) | Automatic retry with exponential backoff (3 attempts) |
| **HTML/Text** | Defaults to plain text; filter to switch | Always HTML with automatic plain-text fallback |
| **Template Override** | `pluggable.php` function replacement | Admin UI template editor + database templates |
| **From Address** | `wp_mail_from` / `wp_mail_from_name` filters | Settings System `email.from_address` / `email.from_name` |
| **Batch/Digest** | Not built-in (requires plugins) | Native batched and digest delivery modes |
| **Rate Limiting** | None | Built-in per-minute and daily limits |
| **Unsubscribe** | Not built-in | Per-category opt-out with unsubscribe links |

## Architecture Overview

### Data Flow

```
CMS Event (e.g., user registers)
  -> Event Dispatcher receives event
    -> Registered email listener handler fires
      -> Handler calls queueEmailForEvent() helper
        -> Helper: fetch template, check unsubscribes, render variables
          -> Insert into emailQueue table (status: "queued")
            -> For immediate: ctx.scheduler.runAfter(0, email.send)
            -> For batched: scheduledFor = Date.now() + batchWindow
            -> For digest: generated by digest cron, not queued individually
              -> email.send action: call Resend API
                -> On success: update status to "sent", emit notification.email_sent
                -> On failure: retry with backoff or mark "failed", emit notification.email_failed
```

### Real-Time Behavior

- **Admin Email Queue Monitor**: Uses `useQuery(api.email.listQueue, ...)` for real-time queue status updates. When an email transitions from "queued" to "sent" or "failed", the admin UI updates automatically.
- **Email Statistics Cards**: Real-time aggregation queries display live counts of sent, failed, queued, and bounced emails.
- **Template List**: Template `totalSent` and `lastSentAt` fields update reactively when emails are sent.
- **User Email Preferences**: Uses `useQuery(api.email.getUserPreferences, ...)` for reactive preference display.

### Authentication & Authorization

- **Admin Operations** (settings, templates, queue, retry): Require Convex Auth authentication + `manage_options` capability (Administrator only).
- **User Preferences** (view/update own): Require Convex Auth authentication + `manage_profile` capability (all roles).
- **System Operations** (queue, send): Called internally by event handlers, not directly by users. No user-facing auth check needed -- these run as internal functions.

### Architectural Position

The Email Notification System is a **consumer** of the Event Dispatcher System. It registers event listeners for 23 specific events (plus 2 digest schedules) and responds by composing and queuing emails. It does not emit its own events directly -- rather, the `email.send` action emits `notification.email_sent` and `notification.email_failed` events through the Event Dispatcher so that the Audit Log can track email delivery.

The system is a **leaf consumer** -- no other system depends on it. Other systems emit events that this system consumes, but they do not call email functions directly.

## Database Schema

### `emailTemplates` Table

Stores the 25 email notification templates. Each template defines the subject, body structure, available variables, and delivery configuration.

```typescript
// convex/schema.ts

const emailPriority = v.union(
  v.literal("immediate"),      // Send as soon as queued
  v.literal("batched"),        // Batch with similar emails (send within 15 min window)
  v.literal("digest"),         // Aggregate into periodic digest (daily/weekly)
);

const emailRecipientType = v.union(
  v.literal("customer"),       // End users (Subscriber, Contributor, Author)
  v.literal("employee"),       // Internal staff (Editor, Author)
  v.literal("admin"),          // Administrators only
  v.literal("custom"),         // Custom recipient list (e.g., all subscribers of a post)
);

emailTemplates: defineTable({
  // --- Identity ---
  slug: v.string(),                               // Unique template identifier (e.g., "welcome-email")
  name: v.string(),                               // Human-readable name (e.g., "Welcome Email")
  description: v.optional(v.string()),            // What this email is for

  // --- Content ---
  subjectTemplate: v.string(),                    // Subject with variables: "Welcome to {site_name}!"
  bodyHtml: v.string(),                           // HTML body template (React Email compiled to HTML)
  bodyText: v.optional(v.string()),               // Plain-text fallback (auto-generated if not set)
  preheaderText: v.optional(v.string()),          // Email preheader/preview text

  // --- Variables ---
  availableVariables: v.array(v.object({          // Variables available in this template
    name: v.string(),                             // Variable name (e.g., "site_name")
    description: v.string(),                      // What this variable contains
    required: v.boolean(),                        // Whether this variable must be provided
    defaultValue: v.optional(v.string()),         // Fallback value if not provided
  })),

  // --- Delivery Configuration ---
  priority: emailPriority,                        // Delivery priority
  recipientType: emailRecipientType,              // Who receives this email
  isActive: v.boolean(),                          // Whether this template is enabled
  eventCode: v.optional(v.string()),              // Event that triggers this email (null for digests)

  // --- Customization ---
  isCustomized: v.boolean(),                      // Whether admin has modified the default template
  defaultSubjectTemplate: v.string(),             // Original subject (for "Reset to Default")
  defaultBodyHtml: v.string(),                    // Original body HTML (for "Reset to Default")

  // --- Metadata ---
  category: v.string(),                           // Template category: "registration", "content", "comment", "security", "system"
  lastSentAt: v.optional(v.number()),             // When this template was last used
  totalSent: v.number(),                          // Lifetime send count

  // --- Timestamps ---
  createdAt: v.number(),
  updatedAt: v.number(),
})
  .index("by_slug", ["slug"])                                     // Lookup by template identifier
  .index("by_event_code", ["eventCode", "isActive"])              // Find active templates for an event
  .index("by_category", ["category"])                             // Templates by category
  .index("by_active", ["isActive"]),                              // Active/inactive filter
```

### `emailQueue` Table

The persistent email queue. Every email to be sent is first inserted here, then processed by the send action. Enables retry, tracking, and auditing.

```typescript
const emailStatus = v.union(
  v.literal("queued"),          // Waiting to be sent
  v.literal("sending"),         // Currently being sent via Resend
  v.literal("sent"),            // Successfully delivered to Resend
  v.literal("delivered"),       // Confirmed delivered to recipient (via webhook)
  v.literal("bounced"),         // Hard bounce (invalid address)
  v.literal("failed"),          // Failed after all retries
  v.literal("cancelled"),       // Cancelled before sending
);

emailQueue: defineTable({
  // --- Recipient ---
  to: v.string(),                                 // Recipient email address
  toName: v.optional(v.string()),                 // Recipient display name
  toUserId: v.optional(v.string()),               // user identifier (if known)

  // --- Sender ---
  from: v.string(),                               // From email address (from settings)
  fromName: v.string(),                           // From display name (from settings)
  replyTo: v.optional(v.string()),                // Reply-to address (if different from "from")

  // --- Content ---
  subject: v.string(),                            // Rendered subject (variables replaced)
  bodyHtml: v.string(),                           // Rendered HTML body
  bodyText: v.optional(v.string()),               // Rendered plain-text body

  // --- Template Reference ---
  templateSlug: v.string(),                       // Which template was used
  templateVariables: v.string(),                  // JSON: variables used to render this email

  // --- Delivery ---
  status: emailStatus,                            // Current delivery status
  priority: emailPriority,                        // Delivery priority
  scheduledFor: v.optional(v.number()),           // When to send (for batched/digest emails)

  // --- Resend Integration ---
  resendId: v.optional(v.string()),               // Resend message ID (after send)
  resendResponse: v.optional(v.string()),         // JSON: full Resend API response

  // --- Retry ---
  attempts: v.number(),                           // Number of send attempts
  maxAttempts: v.number(),                        // Maximum retry attempts (default 3)
  lastAttemptAt: v.optional(v.number()),          // When last attempt occurred
  nextRetryAt: v.optional(v.number()),            // When to retry (if retrying)
  lastError: v.optional(v.string()),              // Error message from last failed attempt

  // --- Tracking ---
  eventId: v.optional(v.id("events")),            // Event that triggered this email
  correlationId: v.optional(v.string()),          // For linking related emails (e.g., bulk sends)

  // --- Timestamps ---
  createdAt: v.number(),                          // When queued
  sentAt: v.optional(v.number()),                 // When successfully sent
  deliveredAt: v.optional(v.number()),            // When confirmed delivered
  openedAt: v.optional(v.number()),               // When recipient opened (if tracked)
})
  .index("by_status", ["status"])                                  // Find queued/failed emails
  .index("by_status_scheduled", ["status", "scheduledFor"])        // Batched emails ready to send
  .index("by_template", ["templateSlug", "createdAt"])             // Emails by template
  .index("by_recipient", ["to", "createdAt"])                      // Emails to a specific address
  .index("by_recipient_user", ["toUserId", "createdAt"])           // Emails to a specific user
  .index("by_resend_id", ["resendId"])                             // Lookup by Resend message ID
  .index("by_event", ["eventId"])                                  // Emails triggered by a specific event
  .index("by_created", ["createdAt"])                              // Chronological queue
  .index("by_retry", ["status", "nextRetryAt"]),                   // Emails due for retry
```

## Actions & Functions

### Mutations

#### `email.queue` - Queue Email

- **Airtable Record:** `recSYCIKbQeQGQyTJ`
- **Convex Function:** `mutations/email.queue`
- **Type:** Mutation
- **Auth:** System-level (called by event handlers, not directly by users)
- **Capabilities:** Administrator (system)
- **Args:**
  ```typescript
  {
    templateSlug: v.string(),                         // Which template to use
    variables: v.string(),                            // JSON: template variables
    recipientEmail: v.string(),                       // Recipient email
    recipientName: v.optional(v.string()),            // Recipient display name
    recipientUserId: v.optional(v.string()),          // user identifier
    eventId: v.optional(v.id("events")),              // Triggering event
    correlationId: v.optional(v.string()),            // For batch correlation
    scheduledFor: v.optional(v.number()),             // Deferred send time
  }
  ```
- **Returns:** `Id<"emailQueue">` or `null` (if template disabled, user unsubscribed, or system disabled)
- **Behavior:**
  1. Validate the email address format.
  2. Fetch the template by `slug` from `emailTemplates` table.
  3. If template `isActive` is `false`, return early.
  4. Check `emailUnsubscribes` for this user + template category (also check "all" category).
  5. Fetch email settings: `from_address`, `from_name`, `reply_to`, `enabled`, `batch_window`.
  6. If `email.enabled` is `false`, return early.
  7. Parse and validate `variables` JSON.
  8. Render the template: replace `{variable_name}` in subject and body.
  9. Inject global variables: `{site_name}`, `{site_url}`, `{current_year}`, `{unsubscribe_url}`, `{recipient_name}`.
  10. Auto-generate plain text from HTML if `bodyText` not set on template.
  11. Calculate `scheduledFor`: immediate = undefined, batched = `Date.now() + batchWindowMs`, digest = return null (digest emails generated by cron).
  12. Insert into `emailQueue` with `status: "queued"`, `attempts: 0`, `maxAttempts: 3`.
  13. For immediate priority, schedule `email.send` via `ctx.scheduler.runAfter(0, ...)`.
  14. Return the queue record ID.
- **Errors:**
  - `VALIDATION_ERROR`: Invalid email address format.
  - `NOT_FOUND`: Template slug does not exist.
  - `TEMPLATE_DISABLED`: Template is inactive.
  - `UNSUBSCRIBED`: Recipient has opted out.
  - `SYSTEM_DISABLED`: Email system master switch is off.
  - `VALIDATION_ERROR`: Missing required template variable.

#### `email.retry` - Retry Failed Email

- **Airtable Record:** `recQziJA1pVrxvMJN`
- **Convex Function:** `mutations/email.retry`
- **Type:** Mutation
- **Auth:** Required
- **Capabilities:** `manage_options` (Administrator only)
- **Args:**
  ```typescript
  {
    queueId: v.id("emailQueue"),            // Email to retry
  }
  ```
- **Returns:** `{ success: true }`
- **Behavior:**
  1. Authenticate user via auth identity.
  2. Check capability: `manage_options`.
  3. Fetch the queue record.
  4. Validate the record is in `"failed"` status.
  5. Reset: `status: "queued"`, `attempts: 0`, `lastError: undefined`, `nextRetryAt: undefined`.
  6. Schedule `email.send` action immediately.
  7. Return success.
- **Errors:**
  - `UNAUTHORIZED`: User not authenticated.
  - `FORBIDDEN`: User lacks `manage_options`.
  - `NOT_FOUND`: Queue record does not exist.
  - `INVALID_STATE`: Queue record is not in `"failed"` status.

#### `email.send` - Send Email via Resend

- **Airtable Record:** `recPNhA7LwszR53lQ`
- **Convex Function:** `actions/email.send`
- **Type:** Action (calls external Resend API)
- **Auth:** System-level (internal)
- **Args:**
  ```typescript
  {
    queueId: v.id("emailQueue"),            // Email queue record to send
  }
  ```
- **Returns:** `void`
- **Behavior:**
  1. Fetch the email queue record by ID.
  2. If `status` is not `"queued"` and not `"sending"`, return.
  3. Update status to `"sending"`.
  4. Check rate limits:
     a. Per-minute: query recent sends (last 60 seconds). If exceeds `email.rate_limit`, reschedule.
     b. Daily: query today's total sends. If exceeds `email.daily_limit`, mark failed.
  5. Call Resend API:
     ```typescript
     const resend = new Resend(process.env.RESEND_API_KEY);
     const response = await resend.emails.send({
       from: `${record.fromName} <${record.from}>`,
       to: record.to,
       subject: record.subject,
       html: record.bodyHtml,
       text: record.bodyText,
       reply_to: record.replyTo,
       headers: {
         "List-Unsubscribe": `<${unsubscribeUrl}>`,
         "List-Unsubscribe-Post": "List-Unsubscribe=One-Click",
       },
     });
     ```
  6. On success: update status to "sent", set `sentAt`, `resendId`, `resendResponse`. Update template `lastSentAt` and increment `totalSent`. Emit `notification.email_sent`.
  7. On failure: increment `attempts`. If < maxAttempts, calculate retry delay (`5000 * Math.pow(2, attempts)`), set `nextRetryAt`, schedule `internal.email.processRetry`. If >= maxAttempts, set status "failed", emit `notification.email_failed`.
- **Events:**
  - `notification.email_sent` on success
  - `notification.email_failed` on final failure
- **Errors:**
  - `NOT_FOUND` / `INVALID_STATE` / `RATE_LIMITED` / `RESEND_ERROR` / `NETWORK_ERROR`

## Edge Cases & Gotchas

1. **Template disabled by admin:** If an admin disables a template, the event handler still fires but `queueEmailForEvent()` returns `null` without queuing. No error is thrown. This is intentional -- admins should be able to disable notifications without touching code.

2. **User has no email:** If a Convex Auth user record has no email address (edge case with OAuth), the email is skipped. A warning is logged but no error event is emitted.

3. **Resend API key not configured:** If `RESEND_API_KEY` environment variable is missing, the `email.send` action fails immediately with a descriptive error. Emails remain queued and can be sent once the key is configured.

4. **Bulk operations flood:** When 100 posts are published at once (bulk publish), 100 `post.published` events fire. The subscriber notification handler should detect bulk operations via `correlationId` and aggregate into a single email: "100 new posts published" rather than 100 individual emails.

5. **Circular event prevention:** The `notification.email_sent` event must NOT trigger any email notifications. The bootstrap listener registration must exclude this event code from all email listener registrations. Without this guard, sending an email would trigger another email indefinitely.

6. **Recipient is the actor:** When an author publishes their own post, they should not receive the "Post Published (Subscribers)" email (if they are also a subscriber). The handler should check `actorId !== recipientUserId`.

7. **Duplicate email prevention:** Before queuing, check if an identical email (same template + same recipient + same event) was sent in the last 5 minutes. Skip if duplicate found.

8. **Long email bodies:** If rendered HTML exceeds 500KB, truncate the content and add a "View all" link to the website.

9. **Invalid email address (synchronous bounce):** If Resend returns a bounce immediately (synchronous validation), mark the queue record as `"bounced"` immediately without retry.

10. **Settings changes during queue processing:** If an admin changes the "from" address while emails are queued, already-queued emails keep their original "from" address (stored at queue time). Only new emails use the updated address. This is by design.

11. **Security-critical emails bypass unsubscribes:** Password reset, password changed, account deactivated, user deletion confirmation, and failed login alerts always send regardless of user unsubscribe preferences. The `queueEmailForEvent()` helper must skip unsubscribe checks for templates in the `SECURITY_CRITICAL_TEMPLATES` list.

12. **Failed login aggregation threshold:** The `onLoginFailed_Admin` handler only sends when 5+ failures occur for the same email within 15 minutes. It should not send on every individual failed login.

13. **Media storage warning deduplication:** The `onMediaUploaded_StorageCheck` handler only sends when storage exceeds 80%, and at most once per 24 hours.

14. **Comment moderation email gating:** The `onCommentCreated_Moderator` handler only sends when site discussion settings require comment moderation. If comments auto-approve, skip this email.

15. **Webhook failure email filtering:** The `onWebhookTriggered_Failure` handler filters `api.webhook_triggered` events, only sending for failed webhooks (status code >= 400 or network error).

16. **New device detection:** The `onLoggedIn_NewDevice` handler checks if the IP/device combination has been seen for this user in the last 30 days. Only sends if the device is new.

17. **Retry-eligible vs non-retryable errors:** Network timeouts, 5xx, and 429 (rate limit) are retryable. 4xx client errors (except 429) and hard bounces are not retryable -- mark as failed immediately.

### Retry Logic

| Attempt | Delay | Formula | Cumulative Wait |
|---------|-------|---------|----------------|
| 1 | 5 seconds | `5000 * 2^0` | 5s |
| 2 | 10 seconds | `5000 * 2^1` | 15s |
| 3 | 20 seconds | `5000 * 2^2` | 35s |
