# ConvexPress Audit Report

Date: 2026-04-06
Scope: `ConvexPress-Website`, `ConvexPress-Admin`
Auditor: Codex

## Audit Criteria

This audit used the following criteria across both apps:

1. Build health
   - Can each app complete a production build from its workspace root?
   - Are there chunk-size warnings or other build-time signals of structural debt?

2. Type health
   - Does `check-types` run at the workspace root?
   - Does the app-level TypeScript pass without local suppressions or broken route types?

3. Routing integrity
   - Do route module IDs match the generated TanStack route tree?
   - Do `navigate()` calls satisfy current route search/params contracts?
   - Are layout/public paths consistent with actual generated public URLs?

4. Architecture boundary compliance
   - Website must remain a Convex consumer.
   - Admin owns schema/functions/deployment.
   - Auth/UI stack should match current project direction, or drift must be called out explicitly.

5. Dependency and framework drift
   - Look for stale auth stack usage, old router APIs, mismatched package versions, and placeholder adapters.
   - Check for outdated or pre-release packages where they increase risk.

6. Shared component contract stability
   - When a shared component API changes, dependent call sites must still compile.
   - Broken imports and renamed props count as regressions.

7. Code quality signals
   - Implicit `any`, unused `@ts-expect-error`, impossible comparisons, and broad `as any` casts.
   - TODOs/HACKs that directly impact live behavior.

8. Documentation vs implementation drift
   - Where the Codex docs and current code diverge, the live code wins, but the mismatch is part of the audit.

## Checks Run

- `bun run check-types` in `ConvexPress-Admin`
- `bun run build` in `ConvexPress-Admin`
- `bun run build` in `ConvexPress-Website`
- `bunx tsc --noEmit` in `ConvexPress-Website/apps/web`
- `bun outdated` in both workspace roots
- targeted `rg` scans for auth stack usage, route APIs, stale imports, TODO/HACK markers, and architecture drift

## Executive Summary

- Both apps build successfully.
- Both apps fail type checking.
- The website has a major routing/API migration problem centered on TanStack Router path/search contracts and newly-added help/support routes.
- The admin app has widespread type debt plus at least one hard broken import and several shared-component API mismatches.
- The website root `check-types` command currently gives a false green because no package actually exposes a `check-types` task.
- The codebase shows significant architecture drift from the current Codex mirror: the website still uses Clerk and `ConvexProviderWithClerk`, and both apps still import `shadcn` styling helpers.

## Findings

### Critical

1. Website route modules use stale `createFileRoute()` IDs that do not match the generated route tree.
   - Evidence:
     - `ConvexPress-Website/apps/web/src/routes/_marketing/help.tsx` declares `createFileRoute("/_marketing/help")`.
     - `ConvexPress-Website/apps/web/src/routes/_marketing/help/index.tsx` declares `createFileRoute("/_marketing/help/")`.
     - `ConvexPress-Website/apps/web/src/routes/_marketing/support/index.tsx` declares `createFileRoute("/_marketing/support/")`.
     - Generated router output maps those routes to public IDs `/help`, `/support`, and `/search`, not the `_marketing`-prefixed IDs.
   - Impact:
     - Type generation and route module contracts are out of sync.
     - Help/support pages are the largest source of current website type failures.
     - Future route changes are likely to keep breaking because the code is fighting the generated route tree.

2. Website navigation/search code is written against older TanStack Router search APIs and now fails type checking broadly.
   - Evidence:
     - `navigate({ to: returnTo })` in auth flows no longer satisfies the current route contract.
     - Search UI passes raw objects like `{ q: trimmed }` where the route types now expect the generated reducer/validated shape.
     - `bunx tsc --noEmit` reports failures in `LoginForm.tsx`, `RegisterForm.tsx`, `SearchForm.tsx`, `SearchFilters.tsx`, `SearchOverlay.tsx`, `SearchSuggestions.tsx`, `dashboard.tsx`, `logout.tsx`, and related files.
   - Impact:
     - The website is not type-safe around routing.
     - Authentication redirects and search interactions are brittle and likely to regress during router upgrades.

3. Website root type-checking is effectively disabled at the workspace level.
   - Evidence:
     - Root script exists: `ConvexPress-Website/package.json` has `"check-types": "turbo check-types"`.
     - App package has no `check-types` script: `ConvexPress-Website/apps/web/package.json`.
     - Running `bun run check-types` at the website root reports `No tasks were executed as part of this run.`
   - Impact:
     - CI or local health checks can report green while the app contains dozens of actual TypeScript failures.
     - This is an audit-process failure, not just a code failure.

### High

4. Website auth stack is still Clerk-based, which conflicts with the current Codex architectural direction and increases migration risk.
   - Evidence:
     - `ConvexPress-Website/apps/web/package.json` depends on `@clerk/clerk-react` and `@clerk/tanstack-react-start`.
     - `ConvexPress-Website/apps/web/src/routes/__root.tsx` wraps the app in `ClerkProvider` and `ConvexProviderWithClerk`.
     - Multiple website routes/components import Clerk hooks directly (`useAuth`, `useClerk`, `useSignIn`, `useSignUp`).
   - Impact:
     - The website is coupled to Clerk at the root-provider level.
     - Any planned move to Convex Auth will be invasive and is currently incomplete.
     - Documentation and implementation are materially out of sync.

5. Admin app has a hard broken package import in `AdminGate`.
   - Evidence:
     - `ConvexPress-Admin/apps/web/src/components/auth/AdminGate.tsx` imports `@convexpress/backend/convex/_generated/api`.
     - The workspace package name is `@convexpress-admin/backend`, not `@convexpress/backend`.
     - `bun run check-types` in admin reports `Cannot find module '@convexpress/backend/convex/_generated/api'`.
   - Impact:
     - This file cannot type-check.
     - It indicates stale package renaming fallout still exists in core auth/bootstrap code.

6. Admin shared-component API changes were not propagated to all call sites.
   - Evidence:
     - `ListTableToolbar` now requires `bulkActionsSlot`.
     - Several call sites still pass only `searchSlot`, producing compile failures in roles/tools tables.
     - `SearchBox` no longer accepts `onSearch`, but at least one consumer still passes it.
   - Impact:
     - Shared admin table flows are partially broken at the type layer.
     - This is the kind of breakage that tends to survive builds but fail during iterative development and refactors.

7. Admin type health is deeply degraded, including frontend and backend packages.
   - Evidence:
     - `bun run check-types` in admin emits a very large error set spanning UI, hooks, routes, and backend Convex modules.
     - Failures include implicit `any`, invalid generics, wrong IDs, missing names, impossible comparisons, and excessive type instantiation in `wordpressSync`.
   - Impact:
     - The admin app currently lacks a trustworthy static-safety baseline.
     - Refactors are higher risk because compiler output is already noisy enough to hide new regressions.

### Medium

8. Both apps still import `shadcn` Tailwind CSS even though the UI components are now primarily Base UI wrappers.
   - Evidence:
     - `ConvexPress-Website/apps/web/src/index.css` imports `shadcn/tailwind.css`.
     - `ConvexPress-Admin/apps/web/src/index.css` imports `shadcn/tailwind.css`.
     - UI components themselves mostly wrap `@base-ui/react`.
   - Impact:
     - Styling stack is muddled.
     - Theme tokens and component assumptions may be carrying compatibility baggage from the old stack.

9. Both builds succeed but ship oversized bundles.
   - Evidence:
     - Website build produced a client chunk around 962 kB for `main`.
     - Admin build produced a client chunk around 3.28 MB for `index` and ~1.01 MB for `EditorLayout`.
     - Both builds emitted Vite chunk-size warnings.
   - Impact:
     - Large JS payloads will hurt first-load performance, especially in admin.
     - This is not an immediate correctness bug, but it is a real operational issue.

10. Website uses a pre-release sanitizer version at the workspace root while the app package resolves a stable version.
   - Evidence:
     - Root website workspace depends on `isomorphic-dompurify` `3.0.0-rc.2`.
     - App package currently shows `^3.0.0` and `bun outdated` reports `3.7.1` available.
   - Impact:
     - Sanitization is security-adjacent; mixed or stale versions are not ideal.
     - This should be normalized and upgraded deliberately.

11. Consumer boundary is documented correctly but still easy to misuse operationally.
   - Evidence:
     - `ConvexPress-Website/packages/backend/README.md` correctly states the website is consumer-only.
     - The website still contains a local `packages/backend` workspace package, which is fine for generated API proxies but easy to misread as an owning backend.
   - Impact:
     - Not a bug by itself.
     - It increases the chance of future architectural mistakes unless naming/docs stay very explicit.

### Low

12. Dependency freshness is uneven but not yet catastrophic.
   - Admin:
     - Tiptap extensions lag several patch/minor releases.
     - `turbo` and `dotenv` are behind current updates.
   - Website:
     - `@icons-pack/react-simple-icons`, `dotenv`, `turbo`, and `isomorphic-dompurify` are behind.
   - Impact:
     - Not the main source of breakage today.
     - Worth addressing after the type/routing baseline is restored.

## Most Important Broken Areas

If this were triaged as a repair sequence, I would treat these as the first buckets:

1. Restore trustworthy type-checking
   - Add a real website `check-types` task in `apps/web`.
   - Reduce admin compiler noise enough that new regressions become visible.

2. Finish the TanStack Router migration on the website
   - Fix stale `_marketing/...` route IDs.
   - Update all `navigate()` and search param calls to current route contracts.
   - Regenerate and align route types after each route change.

3. Clean up obvious admin breakpoints
   - Fix the broken `@convexpress/backend` import.
   - Update all `ListTableToolbar`/`SearchBox` call sites to the current API.

4. Decide the auth direction explicitly
   - Either document Clerk as the current truth and update the Codex docs, or continue the migration and remove Clerk-specific root wiring.

## Residual Risk / Gaps

- I did not execute end-to-end browser flows for every route. This audit is primarily code and build/type based.
- I did not deploy or run Convex mutations against live data.
- Because admin type errors are extremely numerous, the report focuses on root causes and representative failures rather than exhaustively listing every compiler line.

## Raw Outcome

- `ConvexPress-Admin`: build passes, type-check fails badly.
- `ConvexPress-Website`: build passes, app-level type-check fails badly, workspace-level type-check currently gives a false pass.
